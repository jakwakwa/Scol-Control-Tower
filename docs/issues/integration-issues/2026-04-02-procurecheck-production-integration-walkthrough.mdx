# ProcureCheck Production Integration — Walkthrough

## What Changed

The ProcureCheck integration was rebuilt from the ground up. The old implementation used a demo client (`lib/procurecheck-demo.ts`) with mock data and a feature flag (`PROCURECHECK_MOCK_ENABLED`). The new implementation calls the **real ProcureCheck Web API v5** through the Inngest Stage 3 workflow, persists structured per-category results, and renders them with proper loading and error states.

**Before**: Mock data in the frontend, no real API calls, flat procurement payload.

**After**: Full API integration with JWT auth, 6 category checks, structured persistence, and production-grade error handling.

## API Client (`lib/procurecheck/client.ts`)

### Authentication

- `authenticate()` calls `POST /authenticate` with username/password
- Returns a JWT token, cached in-memory for 50 minutes (API TTL is 60 min)
- Token is automatically refreshed on expiry
- `clearTokenCache()` available for testing

### Vendor Creation

- `createVendor(params)` calls `POST /vendors?processBeeInfo=false`
- Maps entity types: `company` → `"4"`, `sole_prop` → `"2"`, `trust` → `"17"`
- Accepts: `vendorName`, `registrationNumber`, `idNumber`, `vatNumber`, `applicantId`

### Polling

- `pollUntilReady(vendorId, opts?)` polls `GET /vendorresults?id=` with exponential backoff
- Intervals: 1s → 2s → 4s → 8s (capped)
- Default timeout: 60 seconds
- Resolves when `ExecutionStatus` is `"Complete"` or `"Completed"`

### Category Results

- `getCategoryResult(vendorId, category)` calls `GET /vendorresults/{category}?id=`
- Six categories fetched in parallel: `cipc`, `property`, `nonpreferred`, `judgement`, `safps`, `persal`

### Proxy Support

- `withProcureCheckProxy()` applies `FIXIE_URL` proxy for static IP egress
- Used by the network-check API route for IP allowlisting verification

## Data Types (`lib/procurecheck/types.ts`)

### ProcurementData (persistence shape)

This is what gets stored in `risk_check_results.payload`:

```typescript
{
  vendorId: string;
  vendor: {
    name: string;
    entityNumber: string;
    entityType: string;
    entityStatus: string;
    startDate: string;
    registrationDate: string;
    taxNumber: string;
    withdrawFromPublic: string;
    postalAddress: string;
    registeredAddress: string;
  };
  summary: {
    categories: Array<{
      category: string;
      outstanding: number;
      total: number;
      executed: number;
      review: number;
      status: "CLEARED" | "FLAGGED" | "PENDING";
    }>;
  };
  categories: Array<{
    id: "cipc" | "property" | "restrictedList" | "legal" | "safps" | "persal";
    description: string;
    reviewed: boolean;
    checks: Array<{
      name: string;
      status: "EXECUTED" | "REVIEW" | "PENDING";
      result: "CLEARED" | "FLAGGED" | "UNKNOWN";
    }>;
  }>;
  checkedAt: string;          // ISO 8601
  provider: "procurecheck";   // literal
}
```

## Inngest Integration

In `stage3_enrichment.ts`, the `check-procurement` step:

1. Calls `executeProcurementCheck(applicantId, workflowId)`
2. On success: persists `payload` and `rawPayload` via `updateRiskCheckMachineState`
3. Sends a success notification with category summary

### Error Handling

| Error Type | Behavior |
|------------|----------|
| Auth errors (401/403) | Classified as `manual_required` — no retries, notification sent |
| Transient errors (network, 5xx) | Re-thrown — Inngest step retries automatically |
| DB unavailable | Throws immediately |
| Applicant not found | Throws immediately |

## Frontend (`procurement-section.tsx`)

The component accepts `data: ProcurementData | null` and `status?: SectionStatus`:

### States

| Machine State | UI Render |
|--------------|-----------|
| `pending` / `in_progress` | Skeleton loader (vendor detail + tabs) |
| `failed` / `manual_required` | Error card with AlertTriangle icon |
| `completed` with data | Full procurement section |
| `completed` without data | Empty state message |

### UI Structure

- **Vendor Detail Accordion**: entity name, number, type, status, tax, addresses (closed by default)
- **Tab Bar**: Overall Summary + 6 category tabs
- **Overall Summary Tab**: table with Category, Outstanding, Total, Executed, Review, Status columns
- **Category Tabs**: description, reviewed indicator (thumbs up/down), checks table (Name, Result with CLEARED/FLAGGED/UNKNOWN badges)
