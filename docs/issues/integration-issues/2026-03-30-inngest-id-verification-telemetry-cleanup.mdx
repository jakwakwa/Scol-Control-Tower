# Inngest ID Verification Telemetry Cleanup

## Problem

`autoVerifyIdentity` in `inngest/functions/id-verification.ts` emitted vendor telemetry twice for the same verification call:

1. once inside `step.run("verify-identity-document")`, and
2. once again after the step returned.

The outer telemetry call used a second `verificationStart`, which made `durationMs` inconsistent with the actual verification operation. Error handling was also split across boundaries, with the throw occurring outside the step.

The Inngest trigger contract remained unchanged: event `document/uploaded` with identity filtering by `documentType`.

## Solution

We refactored the function so verification, telemetry, and failure signaling happen in one place: inside `step.run`.

### Changes made

- Removed the outer `verificationStart` and the second `recordVendorCheckAttempt` call.
- Computed `hasError` once with a safe check:
  - `"error" in verificationResult && Boolean(verificationResult.error)`.
- Emitted telemetry exactly once, adjacent to `processIdentityVerification`.
- Passed the returned error into telemetry (`error: ...`) so failure-type inference has context.
- Threw inside `step.run` using a fallback message (`Unknown identity verification error`) when needed.
- Returned only success-shaped result data from the step.

### Final pattern

```ts
const result = await step.run("verify-identity-document", async () => {
  const verificationStart = Date.now();
  const verificationResult = await processIdentityVerification(applicantId, documentId);
  const hasError = "error" in verificationResult && Boolean(verificationResult.error);

  recordVendorCheckAttempt({
    vendor: "document_ai_identity",
    stage: "async",
    workflowId,
    applicantId,
    outcome: hasError ? "transient_failure" : "success",
    durationMs: Date.now() - verificationStart,
    error: hasError ? verificationResult.error : undefined,
  });

  if (hasError) {
    const errorMessage = verificationResult.error
      ? String(verificationResult.error)
      : "Unknown identity verification error";
    throw new Error(`Identity verification failed: ${errorMessage}`);
  }

  return verificationResult;
});
```

## Why this works

- Keeps wall-clock timing and side effects (`Date.now()`, vendor call, telemetry) inside the Inngest step boundary.
- Produces one metric per step attempt instead of duplicate events.
- Ensures retries are driven by step failure semantics, not post-step checks.
- Improves observability by supplying the failure payload to telemetry inference.

## Prevention & Best Practices

1. Keep I/O, timing, and telemetry in the same `step.run` as the operation they measure.
2. Emit telemetry exactly once per attempt and derive outcome from the same `hasError` check used for control flow.
3. Throw inside the step for retriable vendor failures instead of returning error-shaped success payloads.
4. Keep error messages normalized and string-safe for alerting and dashboards.
5. Monitor `vendor_check_attempt` by `vendor=document_ai_identity`, outcome, and duration percentiles.
6. Treat added `documentType` values as contract changes and update identity-type allowlists with tests.
