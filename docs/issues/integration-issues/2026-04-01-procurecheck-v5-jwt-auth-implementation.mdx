# ProcureCheck Web API v5 — Full Production Integration

## Overview

Replaced the demo/mock ProcureCheck integration with a **production-grade service** that calls the real ProcureCheck Web API v5. The implementation:

- **82 new tests** across 4 test files (TDD approach)
- **Structured per-category results** — CIPC, Property, Restricted List, Legal, SAFPS, Persal
- **Inngest Stage 3 wiring** — automatic vendor check during onboarding workflow
- **Clean error handling** — auth errors → manual review, transient errors → Inngest retries
- **Frontend states** — loading skeleton, error card, real data rendering

## Module Structure

```
lib/procurecheck/
├── types.ts      — Zod schemas for API responses + ProcurementData persistence type
├── client.ts     — JWT auth, vendor CRUD, polling with exponential backoff
├── mapper.ts     — mapVendorResultsToPayload() transformer
├── index.ts      — barrel export
└── __tests__/

    ├── client.test.ts  (27 tests)

    └── fixtures/       (9 JSON fixtures)

lib/services/
└── procurecheck.service.ts  — executeProcurementCheck() orchestrator (7 tests)
```

## Data Flow

```
Inngest Stage 3 check-procurement step
  → executeProcurementCheck(applicantId, workflowId)
    → authenticate() [JWT, 50-min cache]
    → createVendor(params) [entity type mapping]
    → pollUntilReady(vendorId) [exponential backoff, 60s timeout]
    → getCategoryResult() × 6 [parallel Promise.all]
    → mapVendorResultsToPayload()
      → ProcurementData [Zod-validated]
        → persisted in risk_check_results.payload
          → mergeProcurement() [build-report-data.ts]
            → ProcurementSection [procurement-section.tsx]
```

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Zod schemas for all API responses | Runtime validation prevents silent data corruption |
| 50-minute token cache (JWT TTL is 60min) | Reduces auth calls without risking expired tokens |
| Exponential backoff polling (1s → 8s cap) | Balances responsiveness with API rate limiting |
| Auth errors → `manual_required` | 401/403 won't resolve with retries — human review needed |
| Transient errors re-thrown | Inngest step retries handle network/5xx naturally |
| No backward compatibility | Clean break from old `cipcStatus`/`taxStatus`/`beeLevel` shape |

## How to Use

### Environment Variables

```bash
PROCURECHECK_USERNAME="..."
PROCURECHECK_PASSWORD="..."
PROCURECHECK_ENV="sandbox"
PROCURECHECK_BASE_URL="https://xdev.procurecheck.co.za/api/api/v1/"
FIXIE_URL="..."  # optional, for static IP egress
```

### Network Check

```bash
curl -H "Authorization: Bearer $CRON_SECRET" \
  http://localhost:3000/api/integrations/procurecheck/network-check
```

### In Code

```typescript
import { executeProcurementCheck } from "@/lib/services/procurecheck.service";

const result = await executeProcurementCheck(applicantId, workflowId);
// result.vendorId, result.payload (ProcurementData), result.rawPayload
```
